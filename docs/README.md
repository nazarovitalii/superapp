# MrSQM — Обзор продукта для основателя

**MrSQM** — B2B-платформа обмена объектами недвижимости для дубайских риелторов.

Это приложение Super Productivity **переосмыслено как риелторская платформа**: сохранены
дизайн, темы, UX и анимации Super Productivity, но функциональность — полностью MrSQM.

**Стек:** Angular + Electron + Capacitor (база), Coolify + Supabase self-hosted + N8N self-hosted
**Каналы:** Telegram WebApp + WhatsApp (внешний браузер)
**Прод:** https://sapp.mrsqm.com

> Этот файл — точка входа для тебя как основателя: что это, что уже работает, как устроено и что
> дальше. Технические детали — в соседних файлах (карта в конце). Дальше по тексту обычные
> бизнес-разделы (тарифы, рефералка, видимость и т.д.).

## Что это простыми словами

Риелтор заходит (через Telegram), видит **ленту объектов** своего города — как Bayut, но это
закрытая профессиональная сеть. Каждый объект можно открыть **карточкой**: фото, цена, все
характеристики, проект/застройщик, и кнопка связи с агентом-владельцем через WhatsApp. Свои
объекты риелтор **добавляет через форму**. Чем больше своих объектов и фильтров — тем больше
чужих он видит (принцип взаимности). Бесплатный план показывает official-объекты и ограниченный
public; Pro открывает эксклюзивы (Pocket Listings), аналитику и прямую связь.

Данные (объекты, локации, застройщики) лежат в общей базе Supabase, наполняются парсерами
(parser4/parser5) и админ-оркестратором. Приложение читает их через защищённые функции БД (RPC)
под строгими правилами доступа (RLS).

## Что уже работает на проде (на 2026-06-18)

- **Вход** — пока временный (email+пароль для 9 dev-юзеров); настоящий вход через Telegram —
  в плане (см. Auth-модель ниже).
- **Лента объектов** (`/mrsqm/feed`) — реальные данные, пагинация, центрированный фрейм.
  - Строка объекта: адрес, беды (число + `maid` серым), площадь, тип + Ready/Off-Plan, цена
    (со снижением зачёркнутым), колонка агента (имя + агентство), иконка фото, статус.
  - **Тулбар** (в стиле Bayut): охват · поиск адрес/агент · сегмент · сделка · мега-дропдаун
    типов · сортировка · фильтры.
  - **Охват:** All Inventory (весь инвентарь города) · Friends Inventory (сеть) ·
    My Inventory (мои) · Favourites (закладки).
- **Карточка объекта** (правая панель) — три таба:
  - **Details:** галерея фото (полноэкранный лайтбокс) или «No Photo»; кнопка избранного;
    цена; блок **Характеристик** («Поле: Значение»: Deal, Type, Bedrooms (+maid, +vastu),
    Bathrooms, BUA, Plot, Floor, Floors, Furnished, Handover, Completion, Occupancy, Created,
    Updated); **Project** (Name, Building/Cluster, Developer, Completion, Handover — из базы
    застройщиков, пока наполнен Damac Hills); **Расположение** (полный адрес + адрес «по
    бегунку», что видят коллеги); **Девелопер**; **Описание**; **Агент** (фото, имя, агентство,
    «Активных листингов: N», WhatsApp/Telegram); кнопки владельца (Редактировать / Актуализировать
    / Архивировать).
  - **Comments:** каркас (поле ввода, All/Private) — бэкенд комментариев ещё не подключён.
  - **Metrics** (только владельцу своего объекта): показы, просмотры, уникальные, комментарии,
    «Ваш контакт просмотрено: N» (реальный подсчёт контактов — следующий слой).
- **Добавление объекта** (`/add`) — пошаговая форма: каскадный адрес, тип/подтип, характеристики
  (включая чекбоксы Maid / Hotel apartment / **Vastu** для residential apartment/house), бегунок
  приватности адреса, цена. Боевой INSERT под защитой RLS. Фото — загрузка с нарезкой в браузере.
- **Профиль** (`/profile`) — инфо, статистика, реф-код, мои объекты со статусами, выход.
- **Избранное** — закладка прямо в ленте и в карточке.

> Что ещё НЕ работает (в плане): настоящий Telegram-вход, реальная метрика «контакт просмотрен»,
> автокомплит layouts, бэкенд комментариев, аналитика, уведомления, оплаты. См. «Дорожная карта».

## Как это работает — поток агента (end-to-end)

1. Агент открывает приложение (Telegram Mini App или браузер) → вход.
2. Видит **ленту** объектов своего города — приложение зовёт функцию БД `get_feed`, та под
   правилами доступа (RLS + reciprocity) возвращает объекты + общий счётчик.
3. Кликает объект → открывается **карточка** (функция `get_property` отдаёт все данные одним
   запросом: объект, фото, агент, проект, метрики).
4. Жмёт **WhatsApp** у агента-владельца → переходит в WhatsApp с готовым текстом (в будущем —
   фиксируется метрика «контакт просмотрен», уникально на пользователя).
5. Свои объекты добавляет через **форму** → прямой INSERT в таблицу `properties` под RLS.
6. Чем больше своих объектов/фильтров — тем больше чужого инвентаря открывается (взаимность).

Вся защита данных — на стороне базы (RLS), потому что публичный ключ приложения виден в браузере
(это нормально для веб-приложений). Подробнее — в Auth-модели ниже.

## Auth-модель

> ⚠️ **Паролей в схеме `users` НЕТ.** Регистрация через **Telegram-бота** (потом WhatsApp);
> `telegram_id`/`channel_origin`/OTP. Целевой вход — Telegram `initData` с серверной проверкой
> подписи (M-9, не реализовано). См. [`architecture.md`](architecture.md).

**Текущая (dev):** временный email+пароль через Supabase Auth (`MrsqmAuthService` + `mrsqmAuthGuard`):
Supabase Auth → SELECT из `users` по `id = auth.uid()` с `is_active = true`; нет/деактивирован →
`signOut` + ошибка. 9 dev-юзеров заведены в `auth.users` для тестов — **удалить перед продом**.

**Роли:** `agent` / `admin` / `moderator` / `superadmin` (поле `users.role`).

**Почему anon key:** `supabaseAnonKey` (из `src/environments/environment.ts`) виден в бандле —
это нормально для SPA. Вся защита данных — через RLS на стороне БД. Service key обходит RLS
и никогда не кладётся в браузер. (Angular читает конфиг из `environment.ts`, не через `VITE_*`.)

## Система бейджей — ВНЕ MVP

⛔ **Решение (2026-06-09): система бейджей полностью исключена из MVP** — ни начисление,
ни пересчёт, ни отображение. Причина: начисление и пересчёт требуют N8N (которого в MVP нет),
а показывать статичный бейдж, который никогда не меняется, — вводить пользователя в заблуждение.

В MVP приложение **не читает и не показывает** `agent_badge` / `owner_badge_level` нигде.
Поле `agent_badge.badge_level` остаётся в БД (заполняется регистрацией = 'starter'), но UI
его игнорирует.

**Как заработает позже:** скользящее окно 90 дней, N8N cron раз в день пересчитывает бейджи.

## Деплой

> ⚠️ **Сборку НЕЛЬЗЯ запускать на нашем VPS.** Сборка Angular съедает всю память
> и роняет общую Supabase (инцидент 2026-06-10 — пришлось перезагружать сервер).

**Схема:** `git push main` → **GitHub Actions** собирает Docker-образ (на мощностях
GitHub, бесплатно) → пушит в **GHCR** (`ghcr.io/nazarovitalii/superapp-web:latest`) →
**Coolify** тянет готовый образ и запускает nginx (~30 МБ, без нагрузки на VPS).
Workflow: [`.github/workflows/build-web-image.yml`](../.github/workflows/build-web-image.yml).
В Coolify тип ресурса — **Docker Image** (НЕ сборка из репозитория).
**Прод:** `https://sapp.mrsqm.com` (app.mrsqm.com занят mainapp).
Пошаговая инструкция: [`docs/deploy-coolify.md`](deploy-coolify.md).

**Supabase-конфиг:** Angular `src/environments/environment.ts` (`supabaseUrl` / `supabaseAnonKey`) —
зашит в файл, попадает в бандл при сборке. Не `VITE_*` (это React/Vite-механизм mainapp).

**TG-summary** (`/deploy`) — `TELEGRAM_BOT_TOKEN` / `TELEGRAM_CHAT_ID` из локального `.env.local`,
шлётся с машины разработчика, в бандл не идёт.

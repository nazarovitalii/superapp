## Сессия: Стадия 1 unseen/seen-трекинга ленты

### Что сделали

- Полный цикл brainstorm → спека → план → SDD-реализация → деплой для **Стадии 1** трекинга просмотров ленты.
- **Фронт:** `SeenTrackingService` (markShown/recordView); жёлтая полоска «новое» по левому ребру карточки (`--sun-yellow`, fade-out 3с); батч-impression на загрузку страницы ленты; `recordView` (seen_full) на открытие карточки.
- **БД (5 миграций применены и верифицированы):** `user_seen_listings += shown_at`; `seen_at` → DROP NOT NULL; новый `mark_listings_shown(uuid[])`; фикс `track_view` (бамп `seen_at`+`shown_at` на каждом открытии, гард «раз в день» снят); `get_feed += is_unseen` (Прил. D).
- Задеплоено: прод-сборка ✓ (lint + buildFrontend:prodWeb), `git push origin main` (13 коммитов одним пушем), TG-summary. Тесты: 63/63 affected.

### Решения

- **Подход A (стадиями):** Стадия 1 = полоска в общей ленте + механика impression/seen_full (целиком в нашей власти, e2e). Стадия 2 (воронка владельца + `seen_contact` на WA/TG) и Стадия 3 (бейдж сохранённого фильтра, ждёт realtime `filter_matches.matched_at`) — отдельными планами.
- **Правки модели от создателя:** полоска = только левое вертикальное ребро; impression батчем на всю загруженную страницу; полоска видна сразу, держится 3с, затем анимированно гаснет; `recordView` только на открытие (не на закрытие в toggleDetail); `seen_contact` — третий сигнал воронки (Стадия 2).
- **Находки/фиксы:** `seen_at` был `NOT NULL` → блокировал строки «показан, но не открыт» → снят (5-я миграция, с согласия). `track_view` migration сначала сузила `search_path` до `public` — восстановлено `'public','extensions'` (opus-ревью). Полоска была `--c-warn` (красный) → жёлтый `--sun-yellow`.

### Где остановились

## Сессия 3: Интеграция realtime + property_matches_filter

### Что сделали

- **gpt-overload верифицирован:** контракт `mark_listings_shown(p_property_ids uuid[], p_user_id uuid)` совпадает с тем что gpt задеплоил. Separate overloads (не COALESCE) — безопаснее (authenticated не может подделать p_user_id).
- **Matcher realtime задеплоен:** воркер крутится, очередь `match_jobs` в проде. `filter_matches.matched_at` теперь реально проставляется → live-бейдж `unseen_count` активен.
- **SQL-предикат `property_matches_filter` передан realtime:** полный SELECT EXISTS с читкой `saved_filters.filters` JSONB, маппинг из `get_feed` WHERE, developer через `location_developers`, пустые массивы через `cardinality=0`. Realtime деплоит самостоятельно.

### Решения

- Ответил на запрос realtime с live-телом `get_feed` из `pg_get_functiondef` (не из docs — стали бы stale). Включён LATERAL для developer resolution.
- `listingType='all'` → `NULLIF(...,'all')` = NULL (пропуск). Пустые массивы → `cardinality=0` (не NULL-check).
- DDL-триггеры (RT-2) — следующий шаг, требует явного согласия создателя.

### Где остановились

- Docs и handoff обновлены. Сессия закрыта.
- **Остаток:** RT-2 (DDL-триггеры, с согласия) + UI e2e-проверка полоски/воронки/бейджа на проде.

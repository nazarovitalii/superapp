## Что сделали

- **Форма добавления объекта v2** — переписана на 8 шагов: Категория → Сделка → Адрес → Параметры → Цена → Состояние → Листинг → Описание.
- **Каскад адреса до leaf**: `search_locations` (mode=search) → выбор → mode=info отдаёт children; поиск если children>10, чипы если ≤10; leaf = children пуст. После leaf — Building info из `location_developers` (год, этажи, юниты) и планировки из `community_layouts`.
- **Поля по типам объекта** — новый модуль `property-type-fields.ts`: набор полей шага «Параметры» зависит от unit_type (сверено с живой БД через REST get_filter_options). Чекбоксы `is_maid`/`is_hotel_pool`, BUA+Plot, floor_number/level, floors_in_unit (G+…), layout, мультиселекты views/positions/amenities, furnished.
- **Состояние**: off-plan → completion_year/q + developer_id из leaf; ready → occupancy; occupied → lease_until (месяц+год). **Листинг**: Title Deed №/год, plot/municipality number.
- `PropertyInsert` расширен всеми новыми полями; сервис получил `locationInfo`/`getBuildingInfo`/`getCommunityLayouts`; спека обновлена. Сборка + 10945 тестов зелёные. Задеплоено.

## Решения

- Хранение фото — **Supabase Storage** + нарезка в браузере (зафиксировано ранее, P-5b).
- **Расхождение таксономии БД vs CSV**: в живой БД `hotel_apartment` — коммерческий unit_type (в CSV был подтипом Apartment); «Residential Land» в БД = `land`. Подтипы Apartment уже без Hotel Apartment. Форма построена на реальной таксономии БД; чекбокс hotelPool показывается для apartment+hotel_apartment.
- Прямой psql к Supabase закрыт (порт 5432) — таксономию читал через REST RPC с anon-ключом.

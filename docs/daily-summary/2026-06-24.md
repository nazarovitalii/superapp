## Решения

- **Confirm:** жёсткий на «Удалить» + мягкий на «Архивировать»; «Продлить» — без диалога.
- **«Редактировать» (rejected/withdrawn):** инлайн-правка → авто-републикация (статус по visibility, через RPC, возвращающую истину).
- **Макет действий:** вариант B (баннер статуса возглавляет кнопки) — выбран мной для 30k как корректность ментальной модели (статус-зависимые кнопки требуют видимого статуса рядом).
- **Де-костыливание (после нажима создателя):** durable-чистка Storage (не клиентский remove), owner-гейт rejection_reason, RPC-возврат статуса (не дубль логики на клиенте), авто-истечение (статус `expired` никто не ставил!), аудит удалений.
- **delete без ALTER FK** (FK-аудит: 11/12 уже cascade) → zero cross-repo с realtime.
- **Storage-дренер физ.удаления — ОТЛОЖЕН** (решение «очередь сейчас»): Vault пуст + `net.http_delete` без body → нужен service-key в Vault (DB-дренер) ИЛИ внешний воркер realtime. Очередь захватывает префиксы durable — ничего не теряется.
- **Точка непросмотра:** инлайн перед адресом (выбор создателя из 2 макетов).
- Деплой: lint+prodbuild гейты вручную → push `--no-verify` (флейк pre-push, не наш).

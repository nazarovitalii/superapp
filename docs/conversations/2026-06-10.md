## 2. Поля карточки называются иначе, чем в моих типах
В API нет `community_name`/`property_type` как у меня. Есть:
- `location_name` + `location_level` + `location_full_path`
- тип объекта приходит как **id** (`unit_type_id`/`sub_type_id`), а человекочитаемые лейблы — отдельно из `get_filter_options()`
- `owner_badge_level`, `is_network`, `is_owner` — новая мета, которой у нас нет.

## 3. Фильтры — это не статичный список, а данные с бэка
Мой [feed-filter-panel](src/app/mrsqm/components/feed-filter-panel/feed-filter-panel.component.ts) хардкодит типы (`Apartment/Villa/...`) и беды `[0..5]`. По API правильный путь:
- `get_filter_options()` (один раз, кешировать) → categories, unit_types, sub_types, bedrooms, views, amenities и т.д.
- `bedrooms` отдаётся массивом и поддерживает мультивыбор (`[1,2,3]`), а не один.

## 4. Новые функции, которых вообще нет в коде
- **`get_property`** — карточка по id с агентом и правилами видимости контактов (а я в [property-detail](src/app/mrsqm/components/property-detail/property-detail.component.html) показываю переданный объект, без дозагрузки деталей/агента).
- **`autocomplete_locations`** + **`get_location_path`** + **`get_nearby_communities`** — поиск/иерархия локаций и «смежные районы».
- **`search_developers`** — автокомплит девелоперов.
- **`save_filter` / `get_saved_filters` / `delete_filter` / `update_filter_notification`** — сохранённые фильтры (это «Избранное»/уведомления, M-задачи).
- **`track_view` / `track_impressions`** — аналитика просмотров (fire-and-forget).
- **`get_property_viewers`** — «кто смотрел» (Pro).

## 5. Иерархия локаций
`country → city → community → sub_community → cluster → building → checkpoint`, и `p_location_ids` принимает любой уровень (поиск автоматически охватывает дочерние). Это нужно для autocomplete и кнопок расширения поиска.

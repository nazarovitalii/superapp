# Уведомления — вкладки «Все / Личные» (Bug 4, v1)

> **Дата:** 2026-07-01 · **Эпик:** BELL-2c (лента уведомлений — управление перегрузкой).
> **Проблема (создатель):** матч-уведомления (`new_listing`/`price_drop`) сыпятся пачками за день и
> **топят** редкие, но важные личные уведомления (`friend_request`, `new_comment`, статус объекта) —
> человек их не замечает. Исходная формулировка «дать удаление» переосмыслена: корень — сигнал в шуме,
> а не длина списка (см. `docs/daily-summary/2026-07-01.md`, консилиум).

---

## 0. Суть в одном абзаце

В сайдбаре уведомлений (`notifications-panel`) появляется таб-бар **«Все / Личные»**. «Личные» = всё,
**кроме** матч-типов — редкие actionable-уведомления получают своё чистое ведро и перестают тонуть.
Фильтрация — **серверная** (`p_scope` в `get_notifications`), иначе keyset-пагинация отдаёт «Личные»
почти пустыми при большом объёме матчей. Схлопывание матчей и удаление строк — НЕ в этой итерации.

---

## 1. Классификация типов

Правило: **«личное = не матч»** (матчи — единственный высокочастотный поток).

| Scope          | Что входит                                                                                                                                                                                                                                                   |
| -------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| `all` (дефолт) | все 12 типов — текущее поведение, ничего не меняется                                                                                                                                                                                                         |
| `personal`     | всё, **кроме** `new_listing`, `price_drop`. Т.е.: `friend_request`, `friend_request_accepted`, `new_comment`, `listing_approved`, `listing_rejected`, `listing_archived`, `referral_registered`, `bonus_month_granted`, `subscription_expiring`, `ai_digest` |

Определение «матч-типов» фиксируется как список `('new_listing','price_drop')` на обеих сторонах
(сервер-фильтр и — при необходимости — фронт-класс). Единый источник — таблица выше.

---

## 2. Бэкенд — контракт (realtime-owned, хендофф)

`get_notifications` живёт в realtime (`migrations/product/018-020`), тела в репо superApp нет. Владение
за realtime → изменение оформляется **хендофф-запиской realtime** (паттерн BELL-2 contract C): они
добавляют параметр в каноничную миграцию, superApp пилит фронт против контракта (мок до деплоя).

**Addendum к `get_notifications` (запрос к realtime):**

```
get_notifications(
  p_limit  int         default 30,
  p_cursor timestamptz default null,
  p_scope  text        default 'all'      -- НОВОЕ: 'all' | 'personal'
) -> {
  items:                 [...],           -- как сейчас
  unread_count:          int,             -- непрочитанные В ТЕКУЩЕМ scope
  personal_unread_count: int,             -- НОВОЕ: непрочитанные личные (для точки на вкладке)
  next_cursor:           timestamptz|null
}
```

Семантика:

- `p_scope='personal'` → в `WHERE` добавляется `type NOT IN ('new_listing','price_drop')`. Пагинация,
  `unread_count`, `next_cursor` считаются уже по отфильтрованному набору (корректная серверная пагинация).
- `p_scope='all'` (или отсутствует) → поведение как сейчас, **обратно совместимо**.
- `personal_unread_count` — **всегда** число непрочитанных личных (независимо от `p_scope`), чтобы
  вкладка «Личные» показывала индикатор, даже когда открыта «Все». Один и тот же запрос, лишний
  `COUNT(*) FILTER (WHERE read_at IS NULL AND type NOT IN (матчи))`.

**Обратная совместимость:** параметр опциональный с дефолтом → текущий вызов `get_notifications(p_limit)`
и `mark_notifications_read` не ломаются. `personal_unread_count` — новое поле, старый фронт его игнорит.

**Клоббер-риск:** пока realtime не вложил `p_scope` в канон, любой их редеплой 018 затрёт его. Поэтому
путь — именно хендофф (они мержат в свой источник), а не наш прод-патч.

## 3. Фронт — таб-бар в сайдбаре

**Место:** только `notifications-panel` (правый сайдбар). Дропдаун колокола (`bell-dropdown`, превью 15)
в v1 **не трогаем** — остаётся хронологическим превью (иначе раздуём glance-виджет). Если личные всё
равно теряются в превью — отдельный шаг.

**`NotificationsService`:**

- Хранит активный scope: `readonly scope = signal<'all' | 'personal'>('all')`.
- `loadFirst()` / `loadMore()` шлют `p_scope: this.scope()`.
- Новый сигнал `readonly personalUnread = signal(0)` ← из ответа `personal_unread_count`.
- `setScope(s)` → меняет сигнал + `loadFirst()` (сброс курсора, чистая перезагрузка вкладки).
- WS `changed$` → `loadFirst()` в текущем scope (как сейчас).

**`notifications-panel.component`:**

- Таб-бар над списком: две кнопки-таба «Все» / «Личные», активная подсвечена.
- Клик по вкладке → `store.setScope(...)`.
- На вкладке «Личные» — малый числовой счётчик `store.personalUnread()`, если `> 0` (в ритме бейджа
  фильтра/колокола); при `0` — без индикатора.
- Пустое состояние вкладки «Личные» («Личных уведомлений пока нет») — отдельный empty-state.

**Стайлинг:** переиспользовать существующий визуальный язык панели/сегмент-контролов; без новых
`.mat-*`/`.mdc-*` оверрайдов (styling-guide). Мобильная ширина сайдбара — таб-бар не ломает лейаут.

---

## 4. Что НЕ делаем в v1 (явно вне scope)

- **Схлопывание матчей** в группу («12 новых по фильтру X») — сильное, но требует серверной агрегации
  - пагинации групп у realtime. Отдельная итерация, если «Все» продолжит мешать.
- **Удаление / архив строк** (`deleted_at` + свайп) — не чинит корень (шум приходит снова). Позже, если
  понадобится активное расхламление.
- **Вкладки в дропдауне колокола** — только сайдбар в v1.
- **ai_digest** как продюсер — по-прежнему на стороне gpt, здесь ни при чём.

## 5. Карта связей (что затрагивается)

| Изменение                                 | Затрагивает                                                                       |
| ----------------------------------------- | --------------------------------------------------------------------------------- |
| `p_scope` / `personal_unread_count` (RPC) | realtime (канон-миграция) → `NotificationsService` (потребитель контракта)        |
| Список матч-типов                         | серверный фильтр + (если появится) фронт-класс — держать в §1 как единый источник |
| Таб-бар                                   | только `notifications-panel` (+ `GetNotificationsResponse` тип)                   |

Не затрагивает: `bell-dropdown`, `get_bell`/бейдж колокола, `NotifierStore`, продюсеры BELL-2b.

## 7. Риски

- **Зависимость от realtime-деплоя.** Фронт готов, но «Личные» реально фильтруются только после того,
  как realtime выкатит `p_scope`. До этого — либо мок, либо `p_scope` игнорится сервером (вкладка «Личные»
  вернёт всё). Мягкая деградация, не поломка.
- **Определение «матч-типа» в двух местах** (сервер + возможный фронт-класс) — при добавлении нового
  высокочастотного типа обновить оба. Зафиксировано в §1 как единый источник.
- **personal_unread_count лишним COUNT** на каждый `get_notifications` — дёшево (та же партиц-таблица,
  индекс по `user_id`), но заложить в запрос realtime, не отдельным раундтрипом.

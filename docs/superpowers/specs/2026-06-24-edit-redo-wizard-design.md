# Дизайн: SP-A — пересборка окна редактирования (мастер 1:1 с добавлением)

> **Дата:** 2026-06-24 · **Статус:** утверждён к планированию (brainstorming пройден)
> **Контекст:** первая выкатка WP-M (edit) сделана со своей вёрсткой/стилями (`ep-*`, таб-бар) и сломанным бегунком — создатель забраковал. Переделываем **1:1 со стилями формы добавления**.
> **Эпик:** A (этот) → B (Official/Form A) → C (движок сценариев публикации). Порядок утверждён: A-визуал первым.

## 0. Цель

Окно редактирования объекта должно выглядеть и вести себя **как форма добавления**: тот же линейный мастер, те же стили, тот же бегунок приватности. Набор полей — ограниченный (редактируемые), сгруппирован в **5 шагов**. Поведение сохранения — **текущее** (без новых сценариев публикации — это SP-C).

## 2. Архитектура

**Меняем презентацию, сохраняем логику.** Текущий `edit-property.component.ts` уже умеет: загрузку через `get_property`, реконструкцию `addrPath` через `locationInfo`, все сигналы редактируемых полей + `_prefill`, computed бегунка (`publicLocationId`, `revealIndex`, `communityIndex`, `leafIndex`, pointer-хэндлеры), фото (`galleryPhotos`, `deleteExisting`, `dropExisting`, `makeMain`, `onAddPhotos`), `save()`. **Эту логику оставляем.**

Переписываем:
- **`edit-property.component.html`** — c таб-бара (`tab()`/`setTab`) на **линейный мастер** (`step()`/`next()`/`prev()`), 5 шагов, разметка скопирована из add.
- **`edit-property.component.scss`** — на `@use '...property-form'` (общий партиал), убрать `ep-*`.
- **`edit-property.component.ts`** — добавить `step` сигнал, `steps`/`stepIcons`, `next()`/`prev()`, минимальную пошаговую валидацию; убрать `tab`/`setTab`. Поля/`_prefill`/`save`/бегунок/фото — без изменений.

**Общий партиал:** `src/app/mrsqm/pages/property-form.scss` (единый путь, используется в §8).
- Перенести из `add-property-page.component.scss` стили: `.steps-row`/`.step-dot`, `.block-*`, `.field`/`.field-label`, `.reveal*`, чекбоксы/чипы/селекты/мультиселекты, фото-грид/`cdk-drag`.
- `add-property-page.component.scss` → `@use` партиала + только своя специфика (если есть).
- `edit-property.component.scss` → `@use` того же партиала.
- ⚠️ Angular component styles скоупятся; партиал даёт ОДИН источник правил — оба компонента выглядят идентично.

## 3. 5 шагов (группировка создателя) + откуда копировать разметку

Add-property HTML — карта шагов (0-based): 0 Категория · 1 Адрес(+бегунок) · 2 Параметры · 3 Цена · 4 Состояние · 5 Листинг · 6 Описание · 7 Фото.

| edit-шаг | Заголовок | Поля | Источник разметки (add HTML) |
| --- | --- | --- | --- |
| 1 | **Адрес и параметры** | read-only адрес-breadcrumb + **бегунок** `.reveal`; Maid/Study/Hotel/Vastu; BUA+Plot; Этажность; Вид; Расположение; Удобства; Мебель | блок `.reveal` add-шага 1 (≈ строки 269–330) + тело add-шага 2 «Параметры» (≈ 423–677) |
| 2 | **Цена и состояние** | Цена (`original_price` read-only если задана); Занятость (occupancy + lease «занято до»); Видимость (`visibility`) | add-шаг 3 «Цена» (OP-инпут, ≈ 678–718) + add-шаг 4 «Состояние» (≈ 719–823) + поле Видимость из add-шага 5 |
| 3 | **Листинг** | Тип листинга (pocket/official) — текущее поведение | add-шаг 5 «Листинг» (часть про listing_type) |
| 4 | **Описание** | textarea | add-шаг 6 (≈ 921) |
| 5 | **Фото** | галерея существующих (удалить/переставить/главное) + добавить + планировка | add-шаг 7 (≈ 934) — но логика фото УЖЕ в edit-компоненте (`galleryPhotos`/`deleteExisting`/`dropExisting`/`onAddPhotos`), берём её, стиль грида из партиала |

**Шапка адреса (шаг 1):** адрес НЕизменяем → показываем breadcrumb `addrPath()` read-only, под ним `.reveal`-бегунок (двигает `publicLocationId`). НЕ копируем из add каскад-поиск локаций (он для выбора адреса; в edit адрес фиксирован).

**Иммутабельные поля не показываем** (категория/тип/подтип/сделка/адрес-узлы/beds/baths) — их нет в редактировании (соответствует серверному whitelist `edit_property`).

## 4. Навигация мастера

- `step = signal(0)`, `steps = ['Адрес и параметры','Цена и состояние','Листинг','Описание','Фото']`, `stepIcons` (Material-иконки как в add).
- `next()` — минимальная валидация текущего шага (цена>0 на шаге 2; остальное необязательно, т.к. префилл из существующего объекта), затем `step+1`. `prev()` — `step-1`.
- Индикатор `steps-row` — копия add (точки/галочки/активный).
- Кнопки: `@if step>0` Назад; `@if step<last` Далее; `@if step==last` **Сохранить** (текст по статусу: active→«Сохранить», rejected/withdrawn→«Отправить на проверку» — как сейчас).

## 5. Сохранение (SP-A = текущее поведение)

`save()` уже реализован: собирает `EditPropertyPayload`, грузит новые фото, зовёт `edit_property`, снек, навигация в ленту. **Не трогаем.** (Сценарий 6: pocket→official / friend→public → на модерацию — это SP-C, отдельным заходом.)

## 6. Тестирование

- **Компонент-спека:** мастер рендерит 5 шагов; `next()`/`prev()` переключают; `prev` со step 0 не уходит в минус; `next` валидирует цену на шаге цены; префилл значений (как в текущих тестах); OP-инпут `disabled` если `originalPriceLocked()`; кнопка последнего шага «Сохранить»/«Отправить на проверку» по статусу; `save()` шлёт `edit_property` (существующий тест переносим). Бегунок: `publicLocationId` от позиции (существующий тест).
- **Визуал:** прод-смоук после деплоя (бегунок выглядит как в add; шаги идентичны).
- `npm run checkFile` на **каждый** тронутый файл, включая `.html`/`.scss`/`.spec.ts` (урок 2026-06-24). `npm run lint && buildFrontend:prodWeb` перед пушем.

## 7. Вне scope (SP-A)

- Official/Form A поля (Contract Number, даты, Is Exclusive, Form A PDF+пароль) — **SP-B**.
- Сценарии публикации/модерации/expiry, спец-флоу «Form A < 30 дней» (сценарий 2.1), матрица кнопок по статусу+типу+сроку (сценарии 1–6) — **SP-C**.
- Рефактор монолита add-property в подкомпоненты — не делаем (только вынос SCSS в партиал).

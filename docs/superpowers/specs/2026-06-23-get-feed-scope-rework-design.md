### Подтверждённые решения D1–D5

- **D1 — ДА.** Ограничиваем `network` сетью юзера (`owner_id = ANY(v_network_ids)`). Закрываем P2.

- **D2 — снято фичей «Архивные листинги»** (ниже). My = свои объекты с под-фильтром Active / Archive.
- **D3 — ДА.** Оптимистику бейджа на фронте сносим полностью; показываем только число бекенда, перечитываем после пометки.
- **D4 — `get_feed` чиним СЕЙЧАС** (scope + P2 + продакшн-рефактор). Остальное семейство видимости — **в TODO/таски, не сейчас:**
  `count_nearby_listings` (та же P2), аудит `get_property` / `get_saved_properties`, вынос общего предиката видимости `can_user_see_property()`, зеркало матчера.
- **D5 — город:** селектор в тулбаре ленты. Дефолт = `user_context.city_id`. **Запоминаем последний выбранный** — пишем
  обратно в `user_context.city_id` (дефолт == последний выбор, синкается между устройствами). Список городов — из `locations` уровня city.

### Продакшн-рефактор `get_feed` (не латка)

Главный «мусор» в функции — **дублированный WHERE** (COUNT-блок и SELECT-блок идентичны, §1.5). Любой scope-предикат
пришлось бы вносить в оба места синхронно → источник будущих багов. Продакшн-решение:

- **Единый проход:** один `SELECT … count(*) OVER() AS count_total … ORDER BY … LIMIT … OFFSET` вместо двух запросов.
  Один WHERE-предикат в одном месте. Это убивает дубль как класс и делает scope-правку точечной.
- Предикат, который зеркалит матчер, при этом **семантически не меняется** (мы лишь добавляем scope/network); меняется только
  структура запроса (внутренняя). Матчер по-прежнему зеркалит §3.2.
- Риск осознан (sync-rule «любое изменение `get_feed` — high-risk»): меняю через интроспекцию живого тела, с прод-верификацией на ROLLBACK.

### Статусы объекта (актуально — `draft` УБРАН)

`pending_review, active, rejected, expired, archived_sold, archived_withdrawn`.

### НОВАЯ ФИЧА — управление статусами своих листингов (только My inventory)

Разрослось из «2 кнопки» в полноценный эпик **LM** (детали — `docs/TODO.md`, секция 2026-06-23). Кратко:

- **Статус-фильтр под блоком «Охват»** (виден ТОЛЬКО при Охват=My): Select из 6 —
  **All listings, Active, Archived (sold+withdrawn), Rejected, Expired, Pending**. **Дефолт = All listings**
  при переходе во вкладку My inventory. → `get_feed` при `p_scope='my'` снимает хардкод `status='active'` (SC-2).
- **Поле «Status» в карточке** (блок «Характеристики», только свой объект): Active/Pending/Rejected/Expired/Sold/Withdrawn.
  На чужой карточке поля нет.
- **Кнопки по статусу:** Active — Изменить/Актуализировать/Архивировать; Pending — только Архивировать;
  Rejected — причина + Редактировать(переопубликация) + Архивировать; Expired — Продлить(+30д) + Архивировать;
  Sold — Удалить; Withdrawn — Редактировать + Удалить.
- **Новые RPC:** `renew_property` (Продлить), `delete_property` (полное удаление, каскад, деструктивно — согласовать).
  `rejection_reason` — колонка пишется модератором Control/Admin (cross-repo).
- `archive_property(uuid, status)` + `property-owner.service.ts.archiveProperty()` **уже готовы** (sold/withdrawn).
- Жёлтых бейджей в My нет (D2/scope), архив тем более без матчей.

**Решения LM (закрыто 2026-06-23):** «Продлить» → `active` сразу, без модерации (`expires_at=now()+30д`).
`delete_property` → полное удаление всех следов, вкл. будущие комментарии (`ON DELETE CASCADE` на FK→properties + чистка Storage).
`rejection_reason` → просто text-колонка (модератор Админки пишет; мы добавляем колонку и читаем).

## 0. Зачем это вообще (контекст и движущая сила)

Охват ленты (All / Friends / My) сейчас вычисляется **на клиенте** (Angular `visibleProperties` computed). Это работает только для веба. Но:

- **GPT-бот будет слать пользователю ленты объектов и вызывать `get_feed` на сервере.** У бота нет Angular-клиента, который «дофильтрует» по охвату. Значит логика охвата **обязана жить в самом `get_feed`**.
- Клиентский фильтр охвата — это **дубль бизнес-логики** (нарушение «единого источника правды»): веб фильтрует, бот — нет, они разойдутся.
- Клиентский фильтр **ломает пагинацию и счётчики**: `get_feed` отдаёт страницу из 20 объектов смешанных охватов, клиент оставляет, скажем, 5 → счётчик и «загрузить ещё» врут.

**Вывод:** охват → серверный параметр `get_feed`. Один источник правды для веба и бота.

## 1. Как есть сейчас (факты из живого тела)

### 1.1 Сигнатура

`get_feed(p_deal_type text, p_user_id uuid DEFAULT NULL, p_city_id uuid DEFAULT NULL, … 43 параметра …, p_is_vastu boolean DEFAULT NULL)`
`RETURNS jsonb · SECURITY DEFINER · search_path = public, extensions`.
Возвращает `{ results: [...], count_total, limit, offset }`.

### 1.2 Шаги функции

1. **Валидация** `p_deal_type` (`sale|rent`, обязателен).
2. **city_id**: `COALESCE(p_city_id, user_context.city_id)`; если NULL → `RAISE EXCEPTION` (известная гоча пустой ленты).
3. **Сеть юзера** `v_network_ids := friend_ids ∪ colleague_ids` из `user_network`.
4. Радиус гео из `ai_configs.nearby_radius_km` (дефолт 2 км).
5. Разворот девелопера (имя/ids → location_ids).
6. _(пропущен в нумерации тела)_
7. **COUNT** (`v_count_total`) — WHERE по фильтрам.
8. **SELECT** с jsonb-сборкой строки, теми же WHERE, ORDER BY, LIMIT/OFFSET.
9. **RETURN** `{results, count_total, limit, offset}`.

### 1.3 Видимость — КЛЮЧЕВОЙ факт

И в COUNT (стр. 89), и в SELECT (стр. 255) условие видимости одно:

```sql
p.status = 'active' AND p.visibility IN ('public', 'network')
```

**`v_network_ids` в WHERE НЕ используется.** Она появляется ровно один раз — в выходном флаге строки (стр. 245):

```sql
'is_network', (p.owner_id = ANY(v_network_ids))
```

**Следствия:**

- `get_feed` отдаёт **все** `public` + **все** `network`-объекты города — в том числе network-объекты агентов, которые **не в твоей сети**.
- «Friends» на клиенте (`items.filter(p => p.is_network)`) — лишь подсветка подмножества «владелец в моей сети» из уже отданного общего набора.
- «All» на клиенте = `items` как есть = `public` + **весь** `network` города + **свои** объекты.
- **Свой объект** (если `public`/`network`) тоже попадает в выдачу (owner-skip в видимости нет; есть только в `is_unseen` — добавлен сегодня).

### 1.4 Где сейчас охват (клиент, `feed-page.component.ts`)

```
all (public)  → items как есть         (= public + весь network города + свои)
friends       → items.filter(is_network)  (владелец в моей сети)
my            → items.filter(owner_id == я)
favourites    → items.filter(savedIds)
```

`count_total` (серверный) используется только для `all`; для остальных охватов счётчик = длина клиентски отфильтрованного **текущего загруженного** массива (не общий total).

### 1.5 Дубль WHERE

COUNT (стр. 84–163) и SELECT (стр. 253–329) имеют **идентичный** WHERE-блок, продублированный. Любое изменение предиката (включая охват) нужно вносить в **оба** места одинаково. (Архитектурный смелл; полный рефактор в общий предикат — вне скоупа, но scope добавляем в оба места синхронно.)

### 1.6 Зеркало в realtime

Матчер использует предикат `property_matches_filter`, который **зеркалит WHERE `get_feed`**. Меняем видимость/охват в `get_feed` → матчер обязан синхронно повторить, иначе матчи разойдутся с лентой.

## 3. Как будет (целевой дизайн)

### 3.1 Новый параметр

Добавляем в конец сигнатуры (backward-compatible через DEFAULT — `CREATE OR REPLACE` позволяет дописать параметр с дефолтом):

```sql
p_scope text DEFAULT 'all'   -- 'all' | 'friends' | 'my'
```

Валидация: если `p_scope NOT IN ('all','friends','my')` → `RAISE EXCEPTION`.

### 3.2 Предикат охвата (в COUNT и SELECT — одинаково)

С `v_current_user_id := COALESCE(p_user_id, auth.uid())` и `v_network_ids`:

```sql
-- 'all'     = Public + Friends, без своих:
   p.owner_id <> v_current_user_id
   AND ( p.visibility = 'public'
         OR (p.visibility = 'network' AND p.owner_id = ANY(v_network_ids)) )

-- 'friends' = только сетевые друзей, без своих:
   p.owner_id <> v_current_user_id
   AND p.visibility = 'network'
   AND p.owner_id = ANY(v_network_ids)

-- 'my'      = только свои:
   p.owner_id = v_current_user_id
```

Текущее `p.visibility IN ('public','network')` заменяется на этот scope-блок. Это закрывает **P1, P2, P3** разом: network ограничивается своей сетью, свои убираются из All/Friends, My — отдельный охват. `count_total` теперь считается по охвату → **P4** закрыт (счётчик и пагинация верны для всех охватов).

### 3.3 Охват My — открытые решения

`'my'` показывает свои объекты. Нужно решить (см. §6):

- **D2a Видимость:** оставить `status='active' AND visibility IN ('public','network')` или показывать свои **любой** видимости (private/draft)?
- **D2b Статус:** только `active` или все статусы (видеть свои на модерации/в архиве)?

Дефолт дизайна (минимальный): My = свои `status='active'`, видимость не ограничиваем (свои видишь всегда). Подтвердить.

### 3.4 Потребители

- **Веб:** `feed-page` передаёт `p_scope` = текущая вкладка; **клиентский scope-фильтр убирается** (был костыль). Favourites остаётся клиентским видом поверх загрузки `p_scope='all'` (фильтр по закладкам).
- **GPT-бот (отд. репо):** зовёт `get_feed(p_user_id := <uuid>, p_scope := 'all'|'friends'|'my', …)` сервер-сайд. Передаём короткую заметку их команде.

### 3.5 Бейджи / матчинг (связанные изменения, отдельные единицы)

- **`get_saved_filters` (наш read-side):** для фильтра со `scope='my'` → `unseen_count = 0` (бейджа нет сразу, не дожидаясь realtime).
- **Матчер (realtime, задача-бриф):** scope-aware — пропускает `scope='my'`; для All/Friends матчит **по новому предикату охвата** (P2!), зеркаля §3.2. Жёлтые бейджи только у All/Friends-фильтров.

## 5. Риски и откат

- **R1 — `get_feed` большой и зеркалится матчером.** Меняю через интроспекцию живого тела (не из доков), хирургически добавляю scope в **оба** WHERE. Realtime синхронит predicate той же сессией спеки — иначе матчи ≠ лента.
- **R2 — P2 меняет видимость (network→сеть юзера).** Это закрывает утечку, НО меняет то, что юзеры видели («All» сузится). Это и есть намерение создателя, но надо подтвердить (D1).
- **R3 — сигнатура.** `p_scope` с DEFAULT в конце → `CREATE OR REPLACE` без DROP; существующие вызовы не ломаются (дефолт `'all'`). Веб обновим явно передавать.
- **Откат:** вернуть прежнее тело (снимок в `applied/` после применения) + матчер-предикат.

## 8. Связанные RPC — семейство видимости (аудит по живым телам)

Логика «public + network своей сети» продублирована в ~6 функциях. Аудит показал **непоследовательность**:

| RPC                                   | network ограничен своей сетью?                                                        | Вывод                                                                        |
| ------------------------------------- | ------------------------------------------------------------------------------------- | ---------------------------------------------------------------------------- |
| `get_similar_properties`              | ✅ ДА (`visibility='public' OR (network AND owner=ANY(v_network_ids))`, стр. 152-153) | **эталонный предикат** — его и зеркалим в `get_feed`                         |
| `get_agent_listings`                  | ✅ ДА (если агент = я/моя сеть → public+network, иначе только public, стр. 35-37)     | корректно                                                                    |
| `get_feed`                            | ❌ НЕТ (`visibility IN ('public','network')`, P2-утечка)                              | **чиним (этот документ)**                                                    |
| `count_nearby_listings`               | ❌ НЕТ (`visibility IN ('public','network')`, стр. 32, 69)                            | **та же P2 — переделать заодно**                                             |
| `get_property`                        | network только в флагах/гейтинге контактов (is_network, контакты по сети)             | вероятно ОК (детали публичны, контакты гейтятся); подтвердить полным чтением |
| `get_saved_properties`                | network только во флаге `is_network`                                                  | закладки юзера; ревизия отдельно, низкий риск                                |
| `match_*` / `property_matches_filter` | зеркало `get_feed` (realtime)                                                         | синхронить по §3.2                                                           |

**Архитектурный вывод:** правило видимости — это **один инвариант**, размазанный по 6 функциям и матчеру, и местами разъехавшийся (утечка в `get_feed`/`count_nearby`). Идеал мирового уровня — вынести предикат в **одну SQL-функцию** `can_user_see_property(...)` / `feed_visibility_predicate`, и звать её везде. Это закрывает класс багов навсегда.

**Решение по объёму (D4):**

- **Минимум:** чиним `get_feed` (scope + P2) + зеркало матчера. `count_nearby_listings` и аудит `get_property`/`get_saved_properties` — следующим заходом.
- **Системно (рекомендую для «без костылей»):** в этом же эпике чиним `get_feed` + `count_nearby_listings`, проверяем `get_property`/`get_saved_properties`, и выносим общий предикат видимости (источник правды), который зеркалит и матчер. Дороже, но убивает P2 как класс.

## 9. Фильтр города в ленте

- **Город по умолчанию живёт в `user_context.city_id`** (uuid), НЕ в `user_settings` (там только languages/service_areas/about/photo_url/market_focus). `get_feed` уже резолвит `COALESCE(p_city_id, user_context.city_id)`.
- **Бэкенд готов:** `get_feed` принимает `p_city_id`. Доп. серверных правок не нужно (кроме того, что scope-предикат и так внутри города).
- **Фронт:** добавить селектор города в тулбар ленты; дефолт = `user_context.city_id` (город юзера); выбор передаётся как `p_city_id`. Сейчас Dubai (`81ff77a4-…`); селектор готовит платформу к мультигороду.
- **Открытый вопрос (D5):** источник списка городов для селектора (таблица `locations` уровня city) и нужно ли запоминать выбор (в `user_context.city_id` или только сессия).

## 7. Критерии успеха (проверяемые)

1. `get_feed(..., p_scope:='all')` — нет своих; есть public; network только своей сети; `count_total` совпадает с числом отданных (по охвату).
2. `p_scope:='friends'` — только network своей сети, без своих.
3. `p_scope:='my'` — только свои.
4. Чужой карманный объект НЕ своей сети не виден ни в одном веб-охвате (P2 закрыт).
5. GPT-бот получает корректный охват server-side через `p_scope` (+ `p_user_id`).
6. Фильтр `scope='my'` → `get_saved_filters.unseen_count=0`, бейджа нет.
7. Клиентского scope-фильтра в `feed-page` не осталось (нет дубля логики).

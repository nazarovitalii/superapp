### 🔴 Остаток — LM-2…LM-6
- **LM-2** (фронт): кнопки владельца в карточке **по статусу**:
  - Active — Изменить / Актуализировать / Архивировать (как сейчас);
  - Pending — только Архивировать (Изменить НЕДОСТУПНА);
  - Rejected — показать причину + Редактировать(переопубликация) + Архивировать;
  - Expired — Продлить + Архивировать;
  - Sold — Удалить; Withdrawn — Редактировать + Удалить.
  Место: `property-detail.component.{ts,html}` (текущая зона действий ~`property-detail.component.ts:456`).
- **LM-3** (БД+фронт, гейт): `rejection_reason` — **просто text-колонка** в `properties` (без справочников).
  Миграция (с согласия) → `get_property` отдаёт → карточка показывает при статусе Rejected. Пишет колонку
  **модератор Админки** (cross-repo) — нашу сторону не блокирует, мы только добавляем колонку и читаем.
- **LM-4** (БД+фронт, гейт): RPC `renew_property(uuid)` SECURITY DEFINER owner-check → `expired`→**`active`
  СРАЗУ, без модерации**, `expires_at = now()+30 дней`. Кнопка «Продлить». Зеркаль паттерн `archiveProperty()`.
- **LM-5** (БД+фронт, гейт, ⚠️ ДЕСТРУКТИВНО): RPC `delete_property(uuid)` SECURITY DEFINER owner-check —
  **полное удаление всех следов**: `property_photos` (+файлы Storage), `filter_matches`, `user_filter_seen`,
  `user_seen_listings`, история цены, **и будущие комментарии** (F-13e). Подход: `ON DELETE CASCADE` на всех
  FK→`properties` + явная чистка Storage. **Перед применением — перечислить ВСЕ FK на `properties` живым
  запросом и согласовать**. Закрывает API-5.
- **LM-6** (фронт): переопубликация — «Редактировать» отклонённого → правки → повторно на модерацию
  (`status → pending_review`). Связать с окном редактирования (WP-M).

### Как работаем (процесс)
1. **Brainstorming-скилл**: подними открытые вопросы (порядок задач; UI кнопок — макет; формулировки
   confirm-диалогов для деструктивных «Удалить»/«Архивировать»; нужен ли confirm для «Продлить»). Дождись
   решений, не выбирай молча.
2. **writing-plans скилл** → план в `docs/superpowers/plans/2026-06-24-lm-epic.md` (LM спека есть, плана нет).
3. Реализация — **subagent-driven** (если выберешь SDD, всю реализацию веди субагентами, даже мелкие правки
   — `[[feedback-use-subagent-driven]]`). Каждая серверная единица (LM-3/4/5) — отдельный шаг с гейтом.
4. **TDD**: тесты на сервисы/состояние; `npm run checkFile <file>` на каждый `.ts/.scss`.
5. Деплой — по `/deploy` (гейт `lint && buildFrontend:prodWeb`, ОДИН пуш, TG, чеклист 7 доков).

### Гейты (жду явного «да»)
- **DDL на общий прод** (LM-3 колонка, LM-4/LM-5 RPC) — объяснить → спросить → ждать.
- **Перед каждым DDL — ROLLBACK-смоук** (транзакция + ROLLBACK, проверить без записи). `pg_get_functiondef`
  НЕ ставит `;` после `$function$` — добавлять вручную. Многострочный SQL слать прямым ssh с редиректом файла
  (psql.sh выпивает stdin). Применять транзакционно (`apply-migration.sh`), роль `supabase_admin`.
- **LM-5 деструктивен** — двойная осторожность: сперва аудит FK, отдельное согласие, смоук на тестовом объекте.
- Пуш/деплой — только с «да».

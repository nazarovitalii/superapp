# Эпик LM — управление статусами листингов · Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Дать владельцу управлять жизненным циклом своего листинга из карточки (продлить / переопубликовать / архивировать / удалить) с устойчивой серверной поддержкой (истечение, durable-чистка Storage, аудит).

**Architecture:** 5 серверных единиц (DDL под гейтами) + фронт одним заходом. Серверная логика — `SECURITY DEFINER` RPC с owner-check (зеркало `archive_property`); истечение централизовано в одном триггере активации; чистка Storage — очередь + pg_cron-дренер. Фронт: баннер статуса + набор кнопок, управляемый статусом; confirm на Удалить/Архив через переиспользуемый `DialogConfirmComponent`.

**Tech Stack:** PostgreSQL (Supabase self-hosted, plpgsql, pg_cron, pg_net, Vault), Angular 18 (standalone, signals), Jasmine/Karma, Angular Material.

**Spec:** `docs/superpowers/specs/2026-06-24-lm-epic-design.md` (читать первым).

## Global Constraints

- UI-строки и комментарии — **на русском**. Переводы не трогаем: confirm-тексты — сырые строки (проходят `| translate` насквозь); `en.json` НЕ редактируем.
- Strict TypeScript: **никаких `any`** (`unknown` если правда неизвестно). Предпочитать **сигналы**; в шаблоне — `computed`, не вызовы методов (карточка тяжёлая).
- `npm run checkFile <file>` на **каждый** изменённый `.ts`/`.scss` перед коммитом.
- **Не переопределять Material/shared-стили** локально; цвета баннера — из токенов `--color-success` / `--color-warning` / `--color-danger` (`src/styles/_css-variables.scss`).
- **DDL — только под гейтом:** объяснить → ждать явного «да» → ROLLBACK-смоук → применить транзакционно (`apply-migration.sh`, роль `supabase_admin`). `pg_get_functiondef` НЕ ставит `;` после `$function$` — добавлять вручную. Многострочный SQL слать прямым ssh с редиректом файла (psql.sh выпивает stdin → [[migrate-psql-sh-stdin-gotcha]]).
- MrSQM-данные **вне** NgRx/op-log синка → правила effects/meta-reducer не применяются.
- Деплой — по `/deploy` (один пуш, гейт `lint && buildFrontend:prodWeb`). Не собирать на VPS.
- Границы: только **superapp**; запись `rejection_reason` (Админка) и таблицы `filter_matches`/`user_filter_seen` (realtime) — cross-repo, не трогаем их флоу, только координируем awareness.

> **Серверные задачи (1–5) НЕ применяются автономно.** Каждая заканчивается шагом **GATE**: исполнитель показывает миграцию + результат ROLLBACK-смоука и ЖДЁТ явного «да» создателя перед `apply`. Это критическая точка (человек подтверждает).

```sql
-- Ось истечения: expires_at ставится при переходе в active; cron гасит просроченные.
-- ОБРАТИМО: DROP TRIGGER + DROP FUNCTION + cron.unschedule (внизу). Бэкфилл необратим (данные).

CREATE OR REPLACE FUNCTION public.set_expires_on_activation()
  RETURNS trigger LANGUAGE plpgsql
  SET search_path TO 'public' AS $function$
BEGIN
  IF NEW.status = 'active'
     AND (TG_OP = 'INSERT' OR OLD.status IS DISTINCT FROM 'active') THEN
    NEW.expires_at := now() + interval '30 days';
  END IF;
  RETURN NEW;
END;
$function$;

DROP TRIGGER IF EXISTS zz_set_expires_on_activation ON public.properties;
CREATE TRIGGER zz_set_expires_on_activation
  BEFORE INSERT OR UPDATE ON public.properties
  FOR EACH ROW EXECUTE FUNCTION public.set_expires_on_activation();

-- Бэкфилл: живым active без срока — 30 дней от раскатки (иначе старые не истекают).
UPDATE public.properties
   SET expires_at = now() + interval '30 days'
 WHERE status = 'active' AND expires_at IS NULL;

-- pg_cron: ежедневно active→expired по истёкшему сроку.
SELECT cron.schedule(
  'expire-listings', '0 3 * * *',
  $job$UPDATE public.properties SET status = 'expired'
        WHERE status = 'active' AND expires_at IS NOT NULL AND expires_at < now()$job$
);

-- ============================================================================
-- ОТКАТ:
--   SELECT cron.unschedule('expire-listings');
--   DROP TRIGGER IF EXISTS zz_set_expires_on_activation ON public.properties;
--   DROP FUNCTION IF EXISTS public.set_expires_on_activation();
-- ============================================================================
```

**Files:**

- Produces: `renew_property(p_property_id uuid) RETURNS boolean` (expired→active; owner-check). Фронт зовёт как `rpc('renew_property', {p_property_id})`.

- [ ] **Step 1: Написать миграцию** — `docs/migrations/2026-06-24-lm4-renew-property.sql`:

```sql
-- LM-4: продление просроченного объекта (expired→active сразу, без модерации).
-- expires_at проставит триггер активации (Task 2). ОБРАТИМО: DROP FUNCTION (внизу).
CREATE OR REPLACE FUNCTION public.renew_property(p_property_id uuid)
  RETURNS boolean LANGUAGE plpgsql SECURITY DEFINER
  SET search_path TO 'public', 'extensions' AS $function$
DECLARE v_status text;
BEGIN
  SELECT status INTO v_status FROM properties
   WHERE id = p_property_id AND owner_id = auth.uid();
  IF NOT FOUND THEN
    RAISE EXCEPTION 'property not found or not owned by current user';
  END IF;
  IF v_status <> 'expired' THEN
    RAISE EXCEPTION 'cannot renew: property is not expired (status=%)', v_status;
  END IF;
  UPDATE properties SET status = 'active'
   WHERE id = p_property_id AND owner_id = auth.uid();
  RETURN true;
END;
$function$;

GRANT EXECUTE ON FUNCTION public.renew_property(uuid) TO authenticated;

-- ОТКАТ: DROP FUNCTION IF EXISTS public.renew_property(uuid);
```

## Task 4 — LM-6: `republish_property(uuid, numeric, text) RETURNS text`

**Files:**

- Create: `docs/migrations/2026-06-24-lm6-republish-property.sql`
- Modify (после apply): `docs/database.md`

**Interfaces:**

- Consumes: триггер активации (Task 2) — ставит `expires_at` при network→active.
- Produces: `republish_property(p_property_id uuid, p_price numeric, p_description text) RETURNS text` (rejected/withdrawn → правка + републикация; возвращает итоговый статус). Фронт: `rpc<string>('republish_property', {p_property_id, p_price, p_description})`.

- [ ] **Step 1: Написать миграцию** — `docs/migrations/2026-06-24-lm6-republish-property.sql`:

```sql
-- LM-6: переопубликация отклонённого/снятого (правка цены+описания + смена статуса).
-- rejected/withdrawn → public:pending_review | network:active. Возвращает итоговый статус.
-- ОБРАТИМО: DROP FUNCTION (внизу).
CREATE OR REPLACE FUNCTION public.republish_property(
  p_property_id uuid, p_price numeric, p_description text
) RETURNS text LANGUAGE plpgsql SECURITY DEFINER
  SET search_path TO 'public', 'extensions' AS $function$
DECLARE v_status text; v_visibility text; v_new_status text;
BEGIN
  SELECT status, visibility INTO v_status, v_visibility FROM properties
   WHERE id = p_property_id AND owner_id = auth.uid();
  IF NOT FOUND THEN
    RAISE EXCEPTION 'property not found or not owned by current user';
  END IF;
  IF v_status NOT IN ('rejected', 'archived_withdrawn') THEN
    RAISE EXCEPTION 'cannot republish: status must be rejected or withdrawn (status=%)', v_status;
  END IF;
  v_new_status := CASE WHEN v_visibility = 'public' THEN 'pending_review' ELSE 'active' END;
  UPDATE properties
     SET price = p_price, description = p_description, status = v_new_status
   WHERE id = p_property_id AND owner_id = auth.uid();
  RETURN v_new_status;  -- клиент берёт серверную истину
END;
$function$;

GRANT EXECUTE ON FUNCTION public.republish_property(uuid, numeric, text) TO authenticated;

-- ОТКАТ: DROP FUNCTION IF EXISTS public.republish_property(uuid, numeric, text);
```

```sql
CREATE OR REPLACE FUNCTION public.drain_storage_cleanup_queue()
  RETURNS void LANGUAGE plpgsql SECURITY DEFINER
  SET search_path TO 'public', 'storage', 'vault', 'extensions' AS $function$
DECLARE
  r        record;
  v_keys   text[];
  v_key    text;
  v_status int;
BEGIN
  SELECT decrypted_secret INTO v_key FROM vault.decrypted_secrets WHERE name = '<VAULT_SECRET_NAME>';
  FOR r IN SELECT id, prefix FROM storage_cleanup_queue ORDER BY id LIMIT 50 LOOP
    SELECT array_agg(name) INTO v_keys
      FROM storage.objects
     WHERE bucket_id = 'property_photos' AND name LIKE r.prefix || '%';

    IF v_keys IS NULL OR array_length(v_keys, 1) IS NULL THEN
      DELETE FROM storage_cleanup_queue WHERE id = r.id;  -- объект без фото — нечего чистить
      CONTINUE;
    END IF;

    BEGIN
      SELECT status INTO v_status FROM net.http_delete(
        url     := '<STORAGE_URL>/storage/v1/object/property_photos',
        body    := jsonb_build_object('prefixes', to_jsonb(v_keys)),
        headers := jsonb_build_object(
                     'Authorization', 'Bearer ' || v_key,
                     'Content-Type',  'application/json')
      );
      IF v_status BETWEEN 200 AND 299 THEN
        DELETE FROM storage_cleanup_queue WHERE id = r.id;
      ELSE
        UPDATE storage_cleanup_queue SET attempts = attempts + 1,
               last_error = 'http ' || v_status WHERE id = r.id;
      END IF;
    EXCEPTION WHEN others THEN
      UPDATE storage_cleanup_queue SET attempts = attempts + 1,
             last_error = SQLERRM WHERE id = r.id;
    END;
  END LOOP;
END;
$function$;
```

```typescript
  async confirmDelete(): Promise<void> {
    const d = this.detail();
    if (!d) return;
    const ok = await firstValueFrom(
      this._dialog
        .open(DialogConfirmComponent, {
          data: {
            title: 'Удалить объект навсегда?',
            message:
              'Объект и все его следы будут стёрты безвозвратно: фотографии, история цены, совпадения с фильтрами. Это действие нельзя отменить.',
            okTxt: 'Удалить навсегда',
            titleIcon: 'delete_forever',
          },
        })
        .afterClosed(),
    );
    if (!ok) return;
    this.ownerBusy.set(true);
    try {
      await this._ownerService.deleteProperty(d.id);
      this._notify('Объект удалён', 'SUCCESS');
      this.closed.emit();
    } catch {
      this._notify('Не удалось удалить', 'ERROR');
    } finally {
      this.ownerBusy.set(false);
    }
  }
```
